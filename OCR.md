# 本地 OCR 页面文字识别设计

## 1. 背景

当前项目已经尝试过基于无障碍服务监听和节点树遍历来识别当前页面元素。

前期验证结果表明：

- 可以稳定拿到前台包名
- 可以拿到部分无障碍文本
- 但很多页面的关键元素并不在无障碍节点树中
- 对自定义控件、自绘文本页面，当前方案无法稳定识别页面关键内容

因此，当前阶段决定弃用“仅依赖无障碍树识别页面”的方案，转而采用“屏幕截图 + 本地 OCR”来获取页面文字信息。

## 2. 目标

接入本地 OCR 能力，周期性识别当前页面中的文字，并将识别结果输出到日志中。

当前目标优先聚焦：

- 当前屏幕能否稳定截图
- 本地 OCR 能否识别当前页面文字
- OCR 结果能否稳定输出到日志
- OCR 命中规则后能否执行最小闭环动作

## 3. 范围

补充说明：

- OCR 页面命中规则的配置文件和字段说明，统一维护在 [OCR_RULES.md](/Users/aschu/IdeaProjects/zz/OCR_RULES.md)

### 3.1 本次要做

- 接入 `MediaProjection` 屏幕截图能力
- 接入本地 OCR 能力
- 获取当前屏幕截图并转为 OCR 输入
- 识别当前页面中的文字
- 将 OCR 结果输出到日志中
- 同时记录当前前台包名
- 支持后台每 2 秒触发一次 OCR
- 支持通过配置文件定义 OCR 关键词命中规则
- 支持命中规则后执行 `WAIT / BACK / SWIPE / CLICK`
- 支持在倒计时这类动态文案上提取 `mm:ss`，并按 `CHANGED / UNCHANGED` 决定是否执行动作
- 支持在 `CLICK` 规则中基于 `num` 做阈值判断，并在两个点击点位之间分支
- OCR 智能模式显示简洁状态浮窗，不显示主倒计时悬浮窗

### 3.2 本次不做

- 不做截图结果持久化
- 不做 OCR 结果结构化分析
- 不做复杂 UI 展示和交互
- 不做基于节点树或局部区域的复杂页面定位
- 不做可视化规则编辑页，当前先使用配置文件维护规则

## 4. 方案概述

当前采用：

- 屏幕截图：通过 `MediaProjection` 获取当前页面 `Bitmap`
- 本地 OCR：使用 Android 本地 OCR 识别页面文字
- 日志输出：将 `pkg + OCR 文字结果` 输出到日志
- 状态展示：通过简洁悬浮窗展示 OCR 最近一次最终状态
- 规则执行：基于默认规则 `assets/ocr_rules_default.csv`、分辨率覆盖规则和外部 override 做匹配与动作执行
- 文本清洗：通过独立清洗配置做整行过滤和字符清洗
- 单字归一化：通过独立混淆字典统一处理 OCR 高频单字错别字
- 调度方式：后台每 2 秒执行一次识别流程

规则文件结构、分辨率覆盖、占位符、`pkg`、`value_policy`、清洗配置和错别字字典的详细说明，统一以 [OCR_RULES.md](/Users/aschu/IdeaProjects/zz/OCR_RULES.md) 为准；这里不再重复展开字段级规则。

整体链路：

1. 用户先完成一次 `MediaProjection` 授权
2. `GestureService` 以前台服务方式开启 OCR
3. 服务侧创建并持有本次 `MediaProjection` / `VirtualDisplay` / `ImageReader` 会话
4. 后台任务每 2 秒触发一次 OCR 流程
5. OCR 直接读取当前截图内容，同时在屏幕上展示简洁状态浮窗
6. 应用从当前截图会话中读取最新屏幕帧
7. 将截图转为 OCR 输入对象
8. OCR 返回识别文本
9. 规则引擎按优先级匹配关键词并决定是否执行动作
10. 将识别结果写入日志，并更新状态浮窗中的最终状态

补充约束：

- 所有外部命令统一通过 `GestureEvent.action` 分发到 `GestureService`
- `Intent.action` 不再作为第二套命令协议使用；额外参数只通过 extras 补充
- `MediaProjection` 授权令牌由服务侧持有，不仅依赖 `Activity` 内存
- 其他自动化场景里的时间参数统一按“主动作停留时长”和“阶段间隔”拆开表达；OCR 后续如需绑定到某个场景，也沿用这套规则
- OCR 运行期间复用现有 `GestureService`
- 当前实现里，`GestureService` 的所有启动路径都会以前台服务方式存活，OCR 是其中一种运行能力
- `MediaProjection` 初始化完成后，当前截图会话会被持续复用，不会在每轮 OCR 时重新创建 `VirtualDisplay`
- 当前实现默认持续持有本次 `MediaProjection` 会话；停止 OCR 轮询时不会主动释放截图会话
- 如果系统主动停止当前 `MediaProjection` 会话，当前实现可能需要用户重新授权

## 5. 技术选型

### 5.1 OCR 方案

优先采用 **Google ML Kit Text Recognition（本地版）**

原因：

- 官方 Android 方案，接入成熟
- 支持本地离线识别
- 可直接处理 `Bitmap`
- 适合当前“先做最小可用闭环”的目标

当前优先选择 bundled 版本，避免首次模型下载导致结果不稳定。

### 5.2 截图方案

本次截图方案明确采用 **MediaProjection**

原因：

- Android 官方屏幕采集能力，可靠性更高
- 不依赖无障碍节点树
- 可以拿到真实屏幕内容，适合识别自定义控件中的文字
- 更符合当前“先稳定拿到页面文字”的目标

实现备注：

- 当前实现已按 Android 14+ 的限制改为“单次授权对应单次截图会话”
- 授权后由服务侧创建一次 `VirtualDisplay`
- 后续 OCR 轮询复用同一套截图会话读取屏幕帧，避免重复创建导致 `SecurityException`

### 5.3 包名来源

当前包名继续使用无障碍服务提供的当前活跃窗口包名：

- 首选：`rootInActiveWindow?.packageName`

原因：

- 当前项目已经有无障碍服务
- 仅用来标注 OCR 结果属于哪个应用
- 不依赖系统任务栈接口

### 5.4 运行宿主

当前 OCR 复用现有 `GestureService`

约束：

- `GestureService` 当前采用“单入口分发模型”：
  - 所有命令统一解析 `GestureEvent.action`
  - OCR 授权结果这类附加数据通过 Intent extras 传递
  - 后续新增命令时，也应优先扩展 `GestureEvent.action`，不要再新增独立的 `Intent.action` 分发
- `ACT_STOP` 作为统一停止入口：当前会同时停止周期任务和当前 OCR，但不主动关闭服务宿主
- OCR 的 2 秒轮询由 `GestureService` 内的 `OcrCoordinator` 调度
- OCR 启动入口会以 `FOREGROUND_SERVICE_TYPE_MEDIA_PROJECTION` 启动 `GestureService`
- 普通手势任务与 OCR 共用同一个服务宿主，但是否绑定 OCR 由具体场景决定
- 当前上滑和点击任务默认都不绑定 OCR；后续新增场景时，再按场景要求决定是否带起 OCR
- 不依赖普通后台服务常驻能力
- 当前也不依赖系统回收后的自动恢复；如果宿主或投屏会话被系统结束，需要由用户重新显式开启
- `MediaProjection` 授权令牌由服务侧持有和消费

## 6. 输出方式

采用“状态浮窗 + 日志输出”：

- 状态浮窗：展示 OCR 最近一次最终状态
- 日志输出：`none` 时保留完整 OCR 文本，命中规则时以规则状态日志为主，错误时保留错误状态；如果某条规则配置了 `log=1`，命中该规则时也会额外打印这一轮 OCR 原始识别文本，便于调试规则

展示内容示例：

```text
ocr: ad_back
```

当前实现会把以 `ocr:` 开头的状态浮窗文本从 OCR 结果中剔除，避免干扰页面规则匹配。
当前匹配前会先做轻量清洗和单字归一化，再交给规则引擎。规则支持别名、动态时间、动态数字、按包名生效范围和点击分支等能力；这些具体配置语义统一在 [OCR_RULES.md](/Users/aschu/IdeaProjects/zz/OCR_RULES.md) 维护。
推荐优先使用这几类状态词：

- `waiting`：OCR 已启动，等待本轮或下一轮执行
- `none`：本轮未识别到有效文字，或未命中任何规则
- `规则 id`：本轮命中某条规则
- `规则 id/skip`：本轮命中了规则，但被动态值比较策略跳过执行
- `规则 id/fail`：本轮命中了规则，但动作执行失败
- `错误码`：本轮识别或截图失败

设计预期中的失败展示示例：

```text
pkg: com.example.xxx
ocr: screenshot_failed
```

或：

```text
pkg: com.example.xxx
ocr: failed
```

当前实现口径：

- 当前阶段优先保留调试型错误码，而不是统一收敛为单一 `failed`
- 为了避免 2 秒轮询时日志刷屏，当前默认只在未命中规则时输出完整 OCR 文本；命中规则时主要记录命中的规则 id
- `CLICK` 规则当前统一输出一套点击决策日志，包含 `rule / screen / extracted / policy / useElseTarget / target`，普通点击和带数值阈值策略的点击共用同一格式
- 匹配规则时会保留完整 OCR 行文本；只有日志展示时才会按单行长度做截断，避免长规则关键词被提前裁掉
- 如果某条规则配置了 `log=1`，则该规则命中时会额外输出完整 OCR 文本，便于按规则定向排查
- 为了便于问题排查，当前日志会输出更细的错误状态，例如：
  - `projection_missing`
  - `projection_create_failed`
  - `screenshot_timeout`
  - `failed:SecurityException`
- 后续如果 OCR 结果展示趋于稳定，再考虑收敛回统一失败文案

## 7. 模块拆分建议

### 7.1 ScreenCaptureProvider

职责：

- 基于 `MediaProjection` 获取当前屏幕截图
- 输出 `Bitmap`

### 7.2 LocalOcrProvider

职责：

- 接收 `Bitmap`
- 执行本地 OCR
- 输出识别文字结果

### 7.3 OcrDebugCoordinator

职责：

- 串起一次完整流程
- 触发截图
- 调用 OCR
- 输出识别日志并触发规则执行

当前实现中该职责已合并到 `OcrCoordinator`

### 7.4 OcrPollingCoordinator

职责：

- 负责后台每 2 秒调度一次 OCR
- 保证同一时间只运行一轮识别
- 失败时输出统一错误状态

当前实现中该职责也由 `OcrCoordinator` 统一承担

## 8. 建议交互

建议保留一个直接入口，例如：

- `开启 OCR`

用户点击后执行以下动作：

- 请求或确认 `MediaProjection` 授权
- 启动 `GestureService` 的 OCR 模式
- 初始化截图会话
- 启动后台定时 OCR 轮询
- 将 OCR 结果输出到日志
- 保持 `GestureService` 的前台服务状态

当前默认后台每 2 秒执行一次 OCR，直到用户手动停止。

## 9. 验收标准

满足以下条件即可认为当前 OCR 能力可用：

- 可以从应用内手动触发一次 OCR
- 可以成功完成 `MediaProjection` 授权
- 可以后台每 2 秒自动执行一次 OCR
- 可以成功获取当前页面截图
- 可以完成本地 OCR 识别
- 可以将识别结果和错误状态输出到日志
- 可以在状态浮窗中看到 OCR 最近一次最终状态
- 在多个页面下测试时，OCR 能输出可读文字结果

按当前实现补充后的验收理解：

- Android 14+ 上，同一次授权后应能在同一截图会话内连续多轮 OCR，而不是每轮都重新授权
- 切换同一应用内页面时，不应因为重复创建截图会话而立即触发 `SecurityException`
- 在用户未显式结束 OCR 的前提下，停止单轮 OCR 轮询不应默认导致重新授权

## 10. 风险与限制

- `MediaProjection` 需要额外授权链路，接入复杂度高于纯无障碍方案
- 后台每 2 秒轮询会带来持续资源消耗
- OCR 运行期间需要以前台服务方式维持稳定性
- Android 14+ 对 `MediaProjection` 限制更严格，截图会话管理不当时容易触发 `SecurityException`
- OCR 结果可能受动画、遮挡、弹窗、字体大小、背景复杂度影响
- 页面文字很多时，浮窗展示可能过长
- 自定义控件虽然能通过截图看到文字，但 OCR 结果仍可能存在误识别
- 当前版本仅提供文字识别结果，不代表已经具备稳定页面识别能力
- 如果系统主动结束当前投屏会话，当前版本可能仍需要用户重新授权
- 如果系统主动结束当前投屏会话，当前实现会停止 OCR，并清理“已开启”状态，等待用户重新授权

## 11. 实施顺序建议

1. 打通 `MediaProjection` 授权和截图链路
2. 接入本地 OCR
3. 加入后台每 2 秒轮询
4. 增加 OCR 日志输出与最小规则执行闭环
5. 将截图实现调整为“单次授权 + 单次截图会话复用”
6. 手工测试多个页面文字识别效果和规则命中效果

## 12. 当前阶段结论预期

如果当前方案稳定可用，可以说明：

- 对于无障碍树拿不到关键元素的页面
- 本地 OCR 仍然有机会拿到页面上的有效文字
- 后续可以继续评估是否基于 OCR 结果做页面识别

如果当前方案仍不稳定，则说明：

- 目标页面即使截图后，文字识别结果仍不稳定
- 后续需要考虑其他方案，例如局部模板匹配或更强的上下文判断
